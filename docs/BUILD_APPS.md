# Building apps with DisSysLab

This guide covers how to design and build a continuously-running
"office" — a small network of agents that ingests data from
sources, processes it through agents, and emits results to sinks.

It assumes you have DisSysLab installed (`pip install -e
'.[dev]'`), an `ANTHROPIC_API_KEY` set in `.env`, and that you've
run at least one gallery office end-to-end (e.g. `dsl run
dissyslab/gallery/my_first_office/`).

The path through this guide:

1. The shape of an app — three files in a folder.
2. Picking your data — what's in the sources/sinks catalogue.
3. Designing the org chart — roles, what they decide, how they hand off.
4. Writing prompts that route correctly.
5. Wiring it up in `office.md`.
6. Running and iterating.
7. Composing offices — splitting into sub-offices.
8. Going further — custom Python roles, custom sinks.

The same example office is used throughout so you can read each
section in isolation.

This guide covers the **by-hand path**: open an editor, type
`office.md`, drop role files in `roles/`. That is the primary path
DisSysLab is optimised for. There is also a *programmatic* path
where you construct an office in Python and call `make_office` to
write the folder for you. That's a separate doc:
[`MAKE_OFFICE.md`](MAKE_OFFICE.md). Both paths produce identical
folders; downstream tools don't care which one made them.

## 1. The shape of an app

A DisSysLab app is a folder with three things:

```
my_office/
├── office.md          # the wiring (what connects to what)
├── roles_lib/
│   └── analyst.md     # one prompt per role
└── (optional) build/
    └── run.py         # generated by `dsl build`; you don't write this
```

`office.md` is plain English describing what the office *is*.
`roles_lib/` holds one prompt file per role. `dsl build` reads
both and produces `build/run.py`, a readable Python file you can
run directly. `dsl run` builds and runs in one step.

Here is the smallest complete app — `dissyslab/gallery/my_first_office/`:

**`office.md`:**

```
# Office: my_first_office

Sources: hacker_news(max_articles=10, poll_interval=600)
Sinks: console_printer

Agents:
Alex is an analyst.

Connections:
hacker_news's destination is Alex.
Alex's briefing is console_printer.
```

**`roles/analyst.md`:** (`roles/` and `roles_lib/` both work; the
v2-recommended name is `roles_lib/`)

```
# Role: analyst

You are a Hacker News analyst. For each story you receive, write
one crisp sentence describing what it's about and why someone
learning software might care.

Send to briefing.
```

That's the entire app. Three blocks of plain English. Every
example below extends this skeleton.

## 2. Picking your data

### Sources you can use today

Sources are the boundary between the outside world and your office.
They poll an external system (an RSS feed, a webhook listener, a
Gmail inbox) and emit one message per item.

The shipped catalogue lives in
`dissyslab/office/utils.py:SOURCE_REGISTRY`:

| Source name | What it pulls |
| --- | --- |
| `hacker_news` | Hacker News RSS feed |
| `bbc_world`, `bbc_tech`, `npr_news`, `al_jazeera`, `mit_tech_review`, `techcrunch`, `venturebeat_ai`, `nasa_news`, `python_jobs` | Various news/tech RSS feeds |
| `weather` | OpenMeteo current-weather snapshot |
| `stocks` | Yahoo Finance ticker prices |
| `bluesky` | BlueSky firehose (live posts) |
| `gmail` | Unread mail from a Gmail inbox |
| `calendar` | Upcoming Google Calendar events |
| `webhook` | HTTP listener (third parties POST to you) |
| `web` | One-shot fetch of a URL via MCP |
| `search` | Brave web search via MCP |

Each source has its own keyword arguments — `max_articles`,
`poll_interval`, `query`, etc. The pattern in `office.md` is
`name(arg=value, arg=value)`:

```
Sources: hacker_news(max_articles=20, poll_interval=300)
Sources: stocks(tickers=["AAPL","TSLA"], poll_interval=120)
Sources: webhook(port=9000)
```

### Sinks you can use today

Sinks are the other boundary — they consume messages and do
something with them (print, save, post, send). The catalogue lives
in `SINK_REGISTRY`:

| Sink name | What it does |
| --- | --- |
| `console_printer` | Pretty-prints to your terminal |
| `intelligence_display` | Richer terminal display (good for briefings) |
| `jsonl_recorder` | Append each message as one JSON line to a file |
| `discard` | Silently drops the message |
| `gmail_sink` | Sends an email |
| `slack_sink` | Posts to a Slack incoming-webhook URL |
| `webhook_sink` | POSTs to an arbitrary URL |
| `mcp_sink` | Calls an arbitrary MCP tool |

Same kw-args pattern. `jsonl_recorder` in particular is your
friend during development:

```
Sinks: jsonl_recorder(path="output.jsonl")
```

That gives you a file you can `cat`, `grep`, or load into a
notebook for analysis.

### When the catalogue isn't enough

If you need a source or sink that isn't on the list, two
options:

- **Quick hack**: in your office's `roles_lib/` write a
  `*.py` role that wires a regular Python class into the
  network directly. Drops you into "Path B" — see section 8.
- **Proper add**: register a new entry in the registries
  (in `dissyslab/office/utils.py`) and place the implementation
  under `dissyslab/components/sources/` or `.../sinks/`. The
  shipped sources are good templates.

## 3. Designing the org chart

The hardest part of building an app is deciding what the agents
should *be*. A good design splits the work into roles where each
role makes one clear decision per message.

### The three common shapes

**Filter.** One role, one input, two outputs (`keep` /
`discard`). Trivial to write, ruthlessly useful. The
`org_news_filter` gallery office is one of these:

```
Agents:
Felix is a filter.

Connections:
al_jazeera's destination is Felix.
Felix's keeper is console_printer.
Felix's discard is jsonl_recorder.
```

`filter` is one role with two `Send to ...` lines in its prompt.
That's all the structure the office needs.

**Handoff.** Two roles in series. The first does triage; the
second writes the polished output. The `news_monitor` sub-office
in `gallery/org_two_office_news/news_monitor/` is exactly this:
Alex (correspondent) decides whether a story is worth analysing,
Morgan (analyst) writes the briefing.

**Feedback loop.** An editor → writer → editor cycle. Used when
the second pass benefits from a first pass's notes. See
`gallery/org_news_editorial/`.

A real app is usually one of these or a small composition of them.
Don't over-design; the right number of agents is "the smallest
number where each one has one clear job."

### What each agent decides

For every agent in your office, you should be able to fill in:

- **Input**: what kind of message arrives.
- **Decision**: what the agent is choosing.
- **Outputs**: which named ports the message can leave on, and
  what each one means.

If you can't articulate one of those three for a role, the role
is poorly defined — either split it into two roles or merge it
into a neighbour.

### Worked example: extending `my_first_office`

Say you want to extend the starter office to filter Hacker News
stories *before* the analyst writes a briefing — only AI/ML
stories make it through, the rest are dropped.

Two-agent design:

- **Felix** (filter): reads each story; decides AI-related or
  not. Outputs: `keep` (forward) or `discard` (drop).
- **Alex** (analyst): receives only the kept stories; writes a
  one-sentence briefing. Outputs: `briefing`.

That maps to:

```
Agents:
Felix is a filter.
Alex is an analyst.

Connections:
hacker_news's destination is Felix.
Felix's keep is Alex.
Felix's discard is discard.
Alex's briefing is console_printer.
```

(Note `discard` on the right of the second connection is a *sink*
— add it to `Sinks:`.)

## 4. Writing prompts that route correctly

Each role file (`roles_lib/<role>.md`) is a prompt. The agent
sends each incoming message through the prompt and the LLM's
reply tells DisSysLab where to route the message next.

The framework reads three things from your prompt file:

1. The `# Role: <name>` header — that's the role's name.
2. Every `Send to <port>` phrase in the body — those are the
   role's *outports*. Order matters; the first one is the
   default.
3. Everything else is the system prompt the LLM sees.

A complete filter prompt:

```
# Role: filter

You receive a news article. Decide whether it's about
artificial intelligence, machine learning, or related research.

If yes, send to keep.
Otherwise send to discard.
```

The two `Send to` lines turn into outports `keep` and `discard`.
The runtime appends a JSON contract to the prompt automatically:

```
Return JSON only, no explanation, no nested JSON:
{"send_to": "<one of: keep, discard>", "text": "<content>"}
```

So you do not write `{"send_to": ...}` instructions yourself —
just say `Send to <port>` for each route and the framework does
the rest.

### Tips that have earned their keep

- **Mention every destination by name in the prose.** "If the
  article is about AI, send to keep; otherwise send to discard"
  routes more reliably than "filter for AI articles, sending the
  rest to discard."
- **Put the routing rule near the end of the prompt.** LLMs
  weight recent text more heavily. The criteria can come earlier;
  the `Send to ...` lines should be the last thing the model reads.
- **For multi-output roles, order matters.** The first `Send to`
  port is the default if the model doesn't return a `send_to`
  field. Make it the most-common outcome.
- **Add fields explicitly when you want them in the output.**
  "Return a `significance` field that is one of CRITICAL, HIGH,
  MEDIUM, LOW" produces structured output downstream agents (or a
  `jsonl_recorder` sink) can use.
- **A role with one output is fine.** Just one `Send to <port>`.
  The agent always routes there; the prompt only has to do
  content work.

### When the LLM goes off-script

If you watch a `jsonl_recorder` output and see messages routed to
the wrong port, three usual causes:

1. **The prompt is ambiguous about when to choose each port.**
   Add the cases-and-defaults explicitly.
2. **The model returned a port name not in your `Send to` list.**
   The framework treats unknown ports as "use the default" — the
   first declared one. Add the spelling the model is using to the
   prompt, or rename your port.
3. **The model returned non-JSON.** Rare with Claude/GPT-4o, more
   common with smaller local models. See `LANGUAGE_MODELS.md` for
   how to tighten the contract.

## 5. Wiring it up in `office.md`

`office.md` has up to six sections. Sections in any order; section
headers are case-insensitive.

| Section | What goes there |
| --- | --- |
| `# Office: <name>` | A header — the office's identifier. |
| `Inputs:` (optional) | External input ports (open-office only). |
| `Outputs:` (optional) | External output ports (open-office only). |
| `Sources:` | One declaration per source. |
| `Sinks:` | One declaration per sink. |
| `Agents:` | One line per agent in the office. |
| `Connections:` | One line per connection statement. |

### Agents

The pattern is `<Name> is a[n] <role>.` or `<Name> is an office at
<path>.` for sub-offices.

```
Agents:
Felix is a filter.
Alex is an analyst.
news_monitor is an office at ../news_monitor.
```

Plural-agreement variation accepted: `Susan and Anna are editors`
parses the same way.

### Connections

The pattern is `<Name>'s <port> is/are <recipient(s)>.` Sources
and external inputs use the literal word `destination` for their
single output port:

```
Connections:
hacker_news's destination is Felix.
Felix's keep is Alex.
Felix's discard is discard.
Alex's briefing is console_printer.
```

For multiple recipients, a single line:

```
Felix's keep is Alex, Morgan and discard.
```

For sub-offices, the recipient names a port on the sub-office:

```
hacker_news's destination is news_monitor's article_in.
news_monitor's article_out is news_editor's article_in.
```

### What the runtime checks

When `dsl build` runs, the runtime's `Network.check()` verifies:

- Every block name is unique.
- Every port mentioned in a connection actually exists.
- Every external input/output you declared is actually wired.
- No connection points at a block that doesn't exist.

If any of those fail, you get an error message naming the file
and the problematic name. The hand-written parser also catches
syntax problems with file + line + snippet.

## 6. Running and iterating

The two commands:

```bash
dsl build my_office/      # parse, validate wiring, emit build/run.py
dsl run my_office/        # build (if stale) + run
```

`dsl run` rebuilds automatically when any source file (any
`office.md` or any file in `roles_lib/` / `roles/`, in your
office or any sub-office) is newer than `build/run.py`. Editing
a prompt and re-running picks up the change without you doing
anything.

The artifact at `build/run.py` is plain executable Python, so
you can also run it directly without going through `dsl`:

```bash
python3 my_office/build/run.py
```

This is exactly what `dsl run` does internally once the build is
up to date. The advantage of `dsl run` is the staleness check;
the advantage of `python3 build/run.py` is that it makes the
"this is just Python you can read and execute" point obvious to
a student.

### When something goes wrong

The most useful debugging trick: open `build/run.py` and read it.
It's plain Python, ~50–150 lines for a typical office. Every
agent, source, sink, and connection is named. Side-of-line
comments on each connection map runtime port names back to your
`office.md` vocabulary:

```python
('Alex', 'out_', 'console_printer', 'in_'),    # Alex's briefing → console_printer
```

A note on the runtime port names: a role with **one** declared
output uses the bare port `out_` (matching Source and Sink
conventions); a role with **multiple** declared outputs uses
indexed `out_0`, `out_1`, … in the order the role's `Send to`
lines appeared. So a filter with `Send to keep` and
`Send to discard` produces ports `out_0` (keep) and `out_1`
(discard).

If a message ends up somewhere unexpected, this file tells you
exactly where. If it doesn't even get there, the problem is in
the role's prompt (LLM is choosing the wrong `send_to`) or in
the connection statement (the wrong recipient name).

### Faster iteration

- Use `console_printer` while developing, switch to
  `jsonl_recorder` when you want to inspect output offline.
- Drop `poll_interval` for development (default behaviour for
  most sources is "poll once and exit"), so each `dsl run`
  produces a finite batch you can examine.
- Add temporary `discard` sinks for branches you don't want to
  see while iterating on another branch.
- `dsl doctor` from inside the office folder confirms your
  `.env`, dependencies, and active backend.

## 7. Composing offices

When an office gets bigger than ~5 agents, splitting into
sub-offices is usually clearer. An *open* office declares
`Inputs:` and `Outputs:` at the top of its `office.md`; a
*closed* (top-level) office has neither and contains its own
sources and sinks.

A two-office network looks like this. Parent office wires three
RSS sources into a sub-office that filters and analyses, then
into a second sub-office that rewrites for publication:

```
# Office: news_org

Sources: al_jazeera(max_articles=5), bbc_world(max_articles=5)
Sinks: intelligence_display

Agents:
news_monitor is an office at news_monitor.
news_editor is an office at news_editor.

Connections:
al_jazeera's destination is news_monitor's article_in.
bbc_world's destination is news_monitor's article_in.
news_monitor's article_out is news_editor's article_in.
news_editor's article_out is intelligence_display.
```

And `news_monitor/office.md` (the open sub-office):

```
# Office: news_monitor

Inputs: article_in
Outputs: article_out

Agents:
Alex is a correspondent.
Morgan is an analyst.

Connections:
article_in's destination is Alex.
Alex's analyst is Morgan.
Morgan's output is article_out.
```

`dsl build news_org/` walks the parent office and recurses into
both sub-offices in one pass. The generated `build/run.py` has
three `build_*()` functions in topological order (children
before parents), with the parent's blocks dict materialising
sub-offices via `build_news_monitor()` calls.

The full example is at `dissyslab/gallery/org_two_office_news/`.

### When to split

- An agent's role file is getting long enough to be confusing.
  Split into two narrower roles.
- You want to reuse a sub-network in another office.
- The same set of agents will run with different sources or
  sinks depending on context.
- You want to test a section of the pipeline in isolation
  (give the open sub-office a tiny test office of its own with
  fixture sources/sinks).

## 8. Going further

Three escape hatches when plain `office.md` + prompt files isn't
enough.

### Custom Python roles

If a role needs deterministic logic the LLM keeps getting wrong
— like extracting an exact field with a regex, or computing a
running average — write the role as a Python file in
`roles_lib/`:

```python
# my_office/roles_lib/extract_ticker.py
from dissyslab.office_v2 import AgentRoleEntry
from dissyslab.blocks.role import Role
import re

_TICKER_RE = re.compile(r"\$([A-Z]{1,5})")

def _build_agent():
    def fn(msg):
        text = msg.get("text", "")
        tickers = _TICKER_RE.findall(text)
        if not tickers:
            return [(msg, "skip")]
        return [({**msg, "tickers": tickers}, "keep")]
    return Role(fn=fn, statuses=["keep", "skip"])

role = AgentRoleEntry(
    name="extract_ticker",
    in_ports=("in_",),
    out_ports=("keep", "skip"),
    factory=_build_agent,
)
```

Then in `office.md`, `Felix is an extract_ticker.` works the
same way as any other agent.

### Custom sinks

If you want output to go somewhere not in the catalogue, write a
class with a `run(msg)` method, drop it under
`dissyslab/components/sinks/`, and add an entry to
`SINK_REGISTRY`. The shipped sinks are good templates —
`console_display.py` is the simplest.

### Using DisSysLab as a Python library directly (Path B)

Everything `office.md` and `dsl build` produce, you can write by
hand using `dissyslab.network.Network`. Skip the office-level
abstractions, construct sources / sinks / `Role` agents directly,
pass them to `Network(blocks=..., connections=...)`, and call
`run_network()`. Useful when you want full control over agent
behaviour — for example, agents that maintain state across
messages or implement custom termination logic.

The `examples/module_*` folders show this style.
